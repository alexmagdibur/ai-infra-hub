Раздел 06
MLOps в банках
Lifecycle · feature store · drift

Модель родилась.
Теперь её надо
кормить.

MLOps — это про то, что происходит с моделью после того, как она «работает на ноутбуке у дата-сайентиста». Без MLOps в банке модель деградирует, никто не знает почему, и это превращается в прямой финансовый убыток. С MLOps — воспроизводимое обучение, авто-деплой, мониторинг, авто-откат.

6.1 Что это

DevOps, но для моделей

MLOps — практики и инструменты, обеспечивающие надёжный жизненный цикл ML-моделей в продакшене. Аналог DevOps, но для моделей. С поправкой на то, что модель — это код + данные + метрики, а не только код.

· Без MLOps

Модель скоринга обучил DS, задеплоили вручную, через полгода она деградировала, никто не знает почему. На разборе ответственный исчез вместе с ноутбуком.

· С MLOps

Воспроизводимое обучение, авто-деплой, мониторинг качества, автоматический откат на предыдущую версию по триггеру. Любой эксперимент можно повторить через год.

6.2 Жизненный цикл

Путь модели в банке

1. ПРОБЛЕМА └─ Бизнес-задача → ML-постановка 2. ДАННЫЕ ├─ Инвентаризация источников (DWH, транзакции, CRM) ├─ Feature engineering └─ Создание датасета → Feature Store 3. РАЗРАБОТКА МОДЕЛИ ├─ Эксперименты (Jupyter, MLflow tracking) ├─ Валидация (backtesting, временные разрезы) └─ Фиксация артефактов (код + данные + метрики) 4. REVIEW И СОГЛАСОВАНИЕ ├─ Техническое ревью (ML-инженер) ├─ Бизнес-валидация └─ Risk / compliance — для кредитных моделей 5. ДЕПЛОЙ ├─ Упаковка (Docker, ONNX) ├─ Staging ├─ Canary 1–5% трафика └─ Full traffic 6. МОНИТОРИНГ — непрерывно ├─ Data drift — изменилось распределение X? ├─ Concept drift — изменилась зависимость Y~X? ├─ Business — модель ещё различает дефолты? └─ Latency — отвечает в SLA? 7. ПЕРЕОБУЧЕНИЕ └─ По расписанию или по триггеру деградации → к п.3
6.3 Компоненты платформы

Пять обязательных подсистем

01 / Feature Store
Централизованное хранилище признаков

Сотни признаков (транзакционные, поведенческие, социальные) с версионированием. Решает проблему train/serve skew — когда признаки при обучении и при inference считаются по-разному.

02 / Tracking
Experiment tracking

MLflow (open-source), Weights & Biases, внутренние разработки. Параметры модели, метрики, артефакты. Любой эксперимент можно воспроизвести.

03 / Registry
Model registry

Каталог обученных моделей с метаданными: версия, метрики, статус (staging / production / archived). Интегрируется с CI/CD пайплайном деплоя.

04 / Serving
Serving infrastructure
  • onlineNVIDIA Triton, TorchServe, FastAPI
  • batchApache Spark + ML-библиотеки
  • a/bмаршрутизация между версиями
05 / Monitoring
Мониторинг — непрерывно, иначе зачем всё это

EvidentlyAI, Prometheus + Grafana, внутренние решения. Метрики: PSI (Population Stability Index), KS-статистика, Gini-коэффициент для скоринга. Без этого слоя любая модель — чёрный ящик в состоянии «надеюсь, работает».

6.4 Банковская специфика

Что отличает MLOps в банке

①
Модельный риск-менеджмент
В отличие от e-commerce, деградация скоринговой модели — прямые финансовые потери. Банк обязан иметь политику модельного риска: независимая валидация, ограничение на автоматическое переобучение без одобрения, документация модели (Model Card).
②
Concept drift в кризис
COVID-19 в 2020, санкционные шоки в 2022 — поведение заёмщиков резко менялось, скоринговые модели на «нормальных» данных теряли качество. Решение: ансамблирование с простыми моделями, ускоренное переобучение, расширение окна данных.
③
Explainability как обязательство
SHAP-значения для каждого решения по кредиту (см. 05). SHAP добавляет задержку → нужен отдельный inference-path или кэширование, и это влияет на дизайн всего слоя serving.
